## Project

"Rota do Sim" — a wedding budget planner (Portuguese/pt-BR UI). Lets a couple track vendors, quotes, installment payments, guest list, and collaborators against a total budget.

## Commands

This project uses **bun** (see `bun.lock`, `bunfig.toml`), not npm/yarn.

- `bun run dev` — start the dev server (vite dev)
- `bun run build` — production build
- `bun run build:dev` — development-mode build
- `bun run preview` — preview a production build
- `bun run lint` — eslint over the whole repo
- `bun run format` — prettier --write over the whole repo

There is no test suite configured in this repo.

## Architecture

### Stack

TanStack Start (file-based SSR React framework) + TanStack Router + TanStack Query, Tailwind CSS v4, shadcn/ui (Radix primitives) in `src/components/ui`, Vite, Nitro (build-only, cloudflare target). The Vite config is provided by `@lovable.dev/vite-tanstack-config` (see `vite.config.ts`) — do **not** manually add the plugins it already documents as included (tanstackStart, viteReact, tailwindcss, tsConfigPaths, nitro, componentTagger, etc.) or the app breaks with duplicate plugins.

This project is synced with [Lovable](https://lovable.dev). Avoid rewriting published git history (force push, rebase/amend/squash of pushed commits) — see `AGENTS.md`.

### Routing (`src/routes/`)

File-based routing via TanStack Router. Read `src/routes/README.md` for the naming conventions before adding routes. Key points:
- `__root.tsx` is the only root layout/shell (wraps `QueryClientProvider` + `DataProvider` + `Toaster`, defines the 404 and error boundary components).
- `_shell.tsx` is a layout route rendering the app chrome (desktop sidebar / mobile bottom nav) around an `<Outlet />`; routes named `_shell.<name>.tsx` render inside it at `/<name>`.
- `index.tsx` (landing) and `onboarding.tsx` sit outside the shell.
- `routeTree.gen.ts` is auto-generated by the TanStack router plugin — never edit it by hand.
- Do not invent `src/pages/`, `src/routes/_app/`, or `app/layout.tsx` (Next.js/Remix conventions) — this app doesn't use them.

### State (`src/data/store.tsx`)

All app data lives in a single React Context (`DataProvider`/`useData`), persisted to `localStorage` (`STORAGE_KEY = "rota-do-sim-state-v1"`) — there is no backend/API. The state shape (`Wedding`, `Category`, `Vendor`, `Installment`, `VendorQuote`, `Guest`, `Collaborator`) is defined and typed here, along with all mutation actions (`addVendor`, `updateInstallment`, `chooseQuote`, etc.) and derived-data hooks like `useTotals` (spend/commitment totals per category). When adding a new domain concept, extend `State` and the `Ctx` interface here rather than introducing parallel local state or another context.

Note `chooseQuote`: accepting a quote creates a new `Vendor` from it and archives every other quote in that same category — quotes and vendors are treated as mutually exclusive per category.

### SSR error handling (`src/server.ts`, `src/start.ts`, `src/lib/error-capture.ts`, `src/lib/error-page.ts`)

There's a deliberate two-layer defense against unhandled SSR errors reaching the client as a raw 500:
1. `start.ts` registers a TanStack Start server middleware that catches request-handler throws and renders `renderErrorPage()` instead.
2. `server.ts` wraps the underlying `@tanstack/react-start/server-entry` fetch handler a second time, because h3 (Nitro's HTTP layer) can swallow in-handler throws into a generic `{"unhandled":true,"message":"HTTPError"}` JSON 500 that a try/catch around the handler won't see. `normalizeCatastrophicSsrResponse` detects that specific shape and substitutes the friendly error page, recovering the real error (if any) via `error-capture.ts`'s global `error`/`unhandledrejection` listeners.

Don't "simplify" this into a single try/catch — the two layers exist for the h3-swallowing case specifically.

Client-side React error boundaries additionally report to Lovable's own error tracking via `src/lib/lovable-error-reporting.ts` (`window.__lovableEvents.captureException`), wired up in `__root.tsx`'s `errorComponent`.

### Styling / design tokens (`src/styles.css`)

Tailwind v4 CSS-based theme (no `tailwind.config.js`) using `@theme inline` + CSS custom properties. Brand colors are defined as OKLCH and exposed as Tailwind color utilities: `sand`, `olive`, `gold`, `graphite`, `terracotta`, `terracotta-soft`, plus the standard shadcn tokens (`background`, `card`, `primary`, `muted`, etc.). Fonts: `font-serif` (Fraunces) for display/headings, `font-sans` (Inter) for body text — both loaded via `@fontsource`.

shadcn/ui config is in `components.json` (style: "new-york", path aliases `@/components`, `@/lib`, `@/ui`, `@/hooks` all resolving under `src/`, matching the `@/*` → `./src/*` path in `tsconfig.json`).

### Formatting helpers (`src/lib/format.ts`)

Currency/date formatting is centralized here (`brl`, `brlShort`, `fmtDate`, `fmtDateShort`, `daysUntil`) — all pt-BR locale aware. Use these rather than ad hoc `Intl`/`Date` formatting when displaying money or dates.
